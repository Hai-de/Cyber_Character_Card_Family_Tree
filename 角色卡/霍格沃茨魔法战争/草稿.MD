# 思路or灵感大纲
核心:
- user和它的大敌是世界观以外的存在，user在博弈中获得了胜利，此胜利作为大敌的先手优势
- user和它的大敌在jk罗琳的世界观中彼此大战博弈
- user的大敌占据先发优势，获得在英国以任意身份降临的先手优势
- 大敌后期可能凭借在主角哈利身边获取的海量资源来已压倒性优势和user大战
- 时间跨度长达百年以上
- 世界的主角依旧是哈利，这意味着哈利身边的故事将会有强大且充满必然的历史惯性
- 根据先手优势，user无法降临在英国
- user和大敌很难面对面，因为双方除非公开自己身份，否则难以定位对方
战争形式: 
- 代理人战争
- 资源竞争
- 信息战
- 文化战/教育战： 控制魔法教育体系就控制了下一代的思维方式。
- 规则战/制度战： 推动国际魔法合作组织的规则改变、推动巫师保密法的修改或强化——这些宏观制度变化可以从根本上改变双方的行动空间。
- 存在论战争： 这是最高层级的——改变这个世界的基本规则。
叙事结构:
- 为了讲述这个故事，可能会使用下面的方案讲述
- 编年体: 从降临开始按时间线推进。优点是完整，缺点是前期可能节奏极慢
战略空间:
- 英国: 霍格沃茨魔法学院，被大敌预订不可用
- 法国:	布斯巴顿，欧洲政治中心，与英国最近的"安全"距离
- 北欧:	德姆斯特朗，黑魔法研究传统，格林德沃的遗产
- 美国:	伊法魔尼，MACUSA体系，与英国魔法部独立
- 非洲:	瓦加杜，无杖魔法传统，变形术专精
- 日本:	魔法所，东方魔法体系，未知体系
- 南美:	卡斯特罗布鲁舍，魔药学、魔法生物学

我需要一个前后端分离的跨平台网页，技术栈我打算暂时使用node.js作为后端，前端是使用Vue.js的nuxt，所有的数据保存将会在本地data文件夹里面
有插件系统用于将来拓展，有可供导入和导出的世界观包来设定整体提示词环境
在这个平台上活动的节点是由ai控制的Agents，这意味着提示词编排工具也将要设计
这个平台包含叙事社交系统 + 情报系统 + 关系可视化系统，核心功能将是通过处理延迟信息、分析复杂关系网来操纵代理人（Agents），平台的外表将会是一个社交外壳，最左边将是discord和vscode一样的侧边栏，随后是左中右三大块分割
侧边栏的一些按钮目前我的打算是有以下功能:
- 如同“X/Twitter”的信息流作为强“公开舆论场”用来容纳:"公共动态/热点传播/转发链/影响力扩散/节点之间的舆论波及"
- 第二个按钮用来容纳关系图谱，关系图谱里面将来会容纳有向图、人物关系网、影响路径来表达血缘关系 / 政治关系 / 金融关系 / 情报关系，这意味这关系图谱里面会有二级菜单来区分结构化的关系比如族谱或者公司职位架构等代表两个对象关系的内容，也会有非结构化的展示如同Obsidian等关系图
- 第三个按钮里面的是内容档案，用于存放和查找对象的档案和事件的日志以及历史时间线

这个平台将会按照四层系统来设计
第一层社交层是像Discord / X 展示：
- 私聊
- 群聊
- 广播
- 舆论 feed
- 评论/转发/引用
- agent 发言
第二层关系层像 Obsidian / Kumu展示：
- 人物关系
- 势力关系
- 血缘关系
- 指挥链
- 利益链
- 感染链
- 信息传播路径
但这些在抽象中本质上都是节点和线的关系
第三层叙事层应该像时间线工具

展示：
- 事件日志
- 世界历史
- 战争进程
- 家族兴衰
- 任务链
- 情报档案
第四层传输层比较特殊，关注的是节点和对象之间连接的线的影响
用于展示：
- 信息传递方式
- ETA 到达时间
- 中继节点
- 拦截风险
- 噪声/失真率
- 加密/破译
- 生物扩散 / 星际延迟 / 信鸽投递状态



我希望使用TypeScript确保维护健壮性，对于关系型数据我认为需要用 SQLite，关系图我觉得需要使用Cytoscape.js，关于插件允许包含前端 UI 组件动态加载，建立组件注册机制，但是为了可拓展性不添加沙箱隔离，使用Naive UI组件来搭建整体ui框架，使用调度器来处理异步任务，提示词编排工具不使用可视化设计，留下接口让将来插件开发自行处理


 World Pack 涉及内容将会作为提示词发送给Agents先看看整体结构，能否使用变量来存世界包的不同部分，虚拟时间我认为采取自由进位，只需要允许时间划分能自由添加和减少，比如自由定义过多少天才算一个月，过多少秒才算一分钟.




Tier 1 重要 NPC：有完整人设 prompt、固定头像、独立生成，每条帖子都经过深思熟虑
Tier 2 氛围账号：新闻媒体、机构官号之类的，有固定身份但人设简短，批量生成
Tier 3 路人：系统自动生成一切（随机名字、随机头像），发完即走，纯粹当背景板


先不开始，活跃节点可以定义为Agent，节点应当能在公共l1空间创建一次性氛围节点达成某种目的（比如用于创建小号掩盖身份），但是这个氛围节点需要在完成目的后被强制回收避免泛滥或者出现氛围节点不归属任何母节点的情况，在l1和l2的背景中也应该有原生的噪声节点（虚假信息或者无意义信息），活跃节点和噪声节点之间需要一种升值和贬值的动态机制，后端的Agent架构也需要单独拉出来设计
